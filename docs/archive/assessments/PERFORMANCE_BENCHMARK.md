# iPig 系統效能基準報告

> **文件版本：** v1.0  
> **測試日期：** 2026-02-25  
> **撰寫日期：** 2026-02-28  
> **工具版本：** Grafana k6 v0.49+

---

## 1. 摘要

iPig 系統後端 API 在 50 個並行虛擬使用者（VU）的階梯式壓力測試下，所有延遲閾值均**通過**：

| 指標 | P95 延遲 | 閾值 | 結果 |
|------|----------|------|------|
| **一般 API**（動物 / 計畫書 / 使用者 / 通知） | **2.31 ms** | < 500 ms | ✅ PASS |
| **報表 API**（庫存 / 總帳 / 採購明細） | **1.76 ms** | < 2,000 ms | ✅ PASS |
| **登入認證** | **30.6 ms** | < 1,000 ms | ✅ PASS |

系統在高峰負載期間維持穩定回應，資料庫連線池未達飽和，健康檢查 100% 通過。

## 3. 測試方法

### 3.1 工具

使用 [Grafana k6](https://k6.io/) 開源負載測試工具，腳本位於 `scripts/k6/load-test.js`。

### 3.2 負載模式

採用**階梯式（ramping）**負載模型，模擬使用者逐步增加的現實場景：

```
VU
50 ┤                    ┌──────┐
   │                    │      │
30 ┤          ┌─────────┘      │
   │          │                │
10 ┤    ┌─────┘                └────┐
   │    │                           │
 0 ┼────┘                           └──
   0   10s   20s   40s   60s   70s
       暖機    負載       高峰    降溫
```

| 階段 | 時長 | 目標 VU |
|------|------|---------|
| 暖機 | 10s | 0 → 10 |
| 負載 | 30s | 10 → 30 |
| 高峰 | 20s | 30 → 50 |
| 降溫 | 10s | 50 → 0 |

**總測試時間：** ~72 秒

### 3.3 涵蓋端點

| 類別 | 端點 | 方法 |
|------|------|------|
| 健康檢查 | `/api/health` | GET |
| 一般 API | `/api/animals` | GET |
| 一般 API | `/api/protocols` | GET |
| 一般 API | `/api/users` | GET |
| 一般 API | `/api/notifications` | GET |
| 報表 API | `/api/reports/stock-on-hand` | GET |
| 報表 API | `/api/reports/stock-ledger` | GET |
| 報表 API | `/api/reports/purchase-lines` | GET |

### 3.4 認證方式

- `setup()` 階段以測試帳號登入一次，取得 JWT Bearer Token
- 所有 VU 共用同一 token，避免大量並行登入觸發 rate limit
- 每個迭代包含：健康檢查 → 4 個一般 API → 3 個報表 API

## 7. 已知限制與注意事項

1. **localhost 測試**：k6 與 API 在同一主機上執行，網路延遲趨近於零。實際生產環境中會增加網路往返時間（RTT），預估增加 1–5 ms。

2. **資料量**：測試時資料庫為開發/測試環境數據量，生產環境隨資料成長可能影響報表查詢效能。

3. **認證共用**：所有 VU 共用同一 JWT token，未測試大量不同使用者同時登入的場景。登入端點受 rate limit 保護（設計如此）。

4. **靜態資源**：本測試僅涵蓋後端 API，不含 nginx 靜態資源服務（前端 SPA）。

5. **寫入操作**：本次僅測試 GET（讀取）端點，未涵蓋 POST/PUT/DELETE 等寫入操作。

## 8. 結論與建議

### 結論

iPig 系統後端 API 在 50 並行使用者的壓力測試中表現優異：

- **所有延遲閾值均大幅通過**，P95 延遲遠低於設定門檻
- **健康檢查與登入 100% 成功**
- **資料庫連線池未達飽和**，仍有擴展空間

### 建議（後續優化方向）

| 優先序 | 建議 | 說明 |
|--------|------|------|
| 1 | 在生產環境重跑基準 | 使用實際硬體與網路環境取得更貼近真實的數據 |
| 2 | 增加寫入操作測試 | 新增動物建立、計畫書提交等 POST 端點的壓測 |
| 3 | 長時間穩定性測試 | 執行 30 分鐘以上的恆定負載測試，觀察記憶體洩漏或效能退化 |
| 4 | 資料量影響評估 | 在百萬筆以上資料量下重跑報表 API 測試 |
