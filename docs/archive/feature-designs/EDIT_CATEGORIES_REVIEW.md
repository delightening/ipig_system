## 2. 業界常見要求 vs 目前實作

### 2.1 已符合

- **權限**：編輯分類與樹 API 使用 `erp.product.edit`，與產品編輯一致。
- **唯讀代碼**：品類／子類代碼不開放修改，避免 SKU 與關聯資料錯亂。
- **軟停用**：以 `is_active` 停用，不刪除資料；停用後 GET /sku/categories 不回傳，新產品無法選用，既有產品仍保留原 code。
- **單一來源**：前端篩選、新增產品、匯入皆用同一組 API，編輯後快取失效即一致。

### 2.2 建議補強（業界常見做法）

| 項目 | 說明 | 目前狀態 | 建議 |
|------|------|----------|------|
| **輸入驗證** | 名稱長度、排序數值範圍、必填。 | ✅ 已實作：後端 `SkuService::update_category` / `update_subcategory` 驗證名稱必填、長度 ≤50、sort_order 0..9999。 | — |
| **稽核紀錄** | 主資料變更應可追溯誰、何時、改什麼。 | ✅ 已實作：PATCH 成功後寫入 `AuditService::log_activity`（ERP / SKU_CATEGORY_UPDATE、SKU_SUBCATEGORY_UPDATE，after_data 含更新後欄位）。 | — |
| **停用前檢查** | 停用品類／子類時，若有產品仍使用，應提示或阻擋。 | 未檢查；停用後產品仍可保留該 code，僅選單不顯示。 | 可選：停用時查詢使用該分類的產品數量，若 > 0 回傳 409 或警告訊息，由前端提示。 |
| **新增／刪除** | 完整 CRUD：新增品類、新增子類、刪除（軟刪已用 is_active）。 | 僅「編輯」：無 POST 品類／子類、無 DELETE。 | 中長期可補 POST；刪除以 is_active=false 已足夠，不一定要實體 DELETE。 |

## 3. UX 與介面

- **每列獨立儲存**：可單獨改一筆就存，不需整表送出，符合業界常見主資料表單。
- **品類／子類階層**：表格分「品類」「子類」列，結構清楚。
- **啟用開關**：用 Switch 表示 is_active，直覺。
- **載入與錯誤**：有 loading、toast 成功／失敗，錯誤訊息依 API 回傳顯示。

小改進可選：每列儲存時僅該列按鈕顯示 loading（目前是任一次 mutation 進行時全部「儲存」按鈕都轉圈），不影響正確性。

## 4. 總結

- **符合度**：與專案文件及「主資料集中管理、代碼與顯示分離、單一 API 來源」等業界做法一致，**已達業界常見水準**。
- **已補強**：  
  1）**後端驗證**：name 必填、長度 ≤50、sort_order 0..9999（`backend/src/services/sku.rs`）。  
  2）**稽核**：品類／子類 PATCH 成功後寫入 `AuditService::log_activity`（ERP / SKU_CATEGORY_UPDATE、SKU_SUBCATEGORY_UPDATE）。  

- **可選後續**：  
  3）**停用前檢查**：若有產品使用該分類，回傳警告或 409。  
  4）「新增品類／子類」API、每列獨立 loading 狀態。
