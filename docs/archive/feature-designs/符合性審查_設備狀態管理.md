## 三、符合性審查結果

### 🔴 嚴重不符合 (Major Non-Conformity)

#### NC-01：設備狀態可被手動覆寫，繞過所有控制流程

**位置：** `backend/src/services/equipment.rs:141-269` (`update_equipment`)

**現狀：** `update_equipment()` 接受 `status` 欄位，允許任何具有 `equipment.manage` 權限的使用者將設備從任意狀態改為任意狀態，無需：
- 審批流程
- 原因說明
- 簽核
- 狀態轉換合法性驗證

**違反條款：**
- **GLP 8.2** — 原始數據完整性：狀態變更缺乏受控流程，可被任意竄改
- **ISO 17025 6.4.4** — 設備停用應有明確程序，不可隨意解除
- **ISO 9001 7.5.1** — 文件化資訊應受控

**風險等級：** 🔴 **高** — 此漏洞使所有其他狀態控制形同虛設

**矯正措施：**
1. 後端 `update_equipment()` 移除 `status` 欄位的接受
2. 前端移除設備編輯 Dialog 的狀態選單
3. 所有狀態變更必須經由對應的業務流程觸發

---

#### NC-02：「閒置」(inactive) 狀態變更無審批流程

**現狀：** `inactive` 狀態無專屬的 UI 入口，也無審批流程。在 GLP 環境中，設備從「啟用」轉為「閒置」意味著該設備不再用於實驗，此決策影響實驗能力與資源配置，必須有審批記錄。

**違反條款：**
- **ISO 17025 6.4.4** — 設備停用需要有程序文件
- **ISO 17025 6.4.13** — 停止使用設備應有文件化決策
- **GLP 4.2** — 設備管理應遵循書面 SOP

**風險等級：** 🔴 **高** — 設備可被無紀錄地閒置，影響實驗室能力評估

**矯正措施：**
1. 建立「閒置申請 → 審批」流程（類似報廢的兩步驟）
2. 申請時需填寫閒置原因
3. 需審批者核准才能變更狀態
4. 恢復啟用同樣需要審批

---

#### NC-03：電子簽名欄位已定義但未實作

**位置：**
- `migrations/013_maintenance_review.sql` — `reviewer_signature_id` 欄位
- `migrations/010_equipment.sql:133-134` — `applicant_signature_id`, `approver_signature_id` 欄位

**現狀：** 資料庫已定義簽名欄位，但 API 請求結構體未包含這些欄位，前端也未實作簽名擷取。

**違反條款：**
- **GLP 8.2** — 簽名是原始數據的一部分
- **ISO 17025 8.4** — 報告需有授權簽名
- **21 CFR Part 11**（若適用 FDA 規範）— 電子紀錄需電子簽名

**風險等級：** 🔴 **高** — 審批動作缺乏不可否認性證據

**矯正措施：**
1. `ReviewMaintenanceRequest` 加入 `signature_id` 欄位
2. `ApproveDisposalRequest` 加入 `approver_signature_id` 欄位
3. 前端審批對話框加入手寫電子簽名元件
4. 未來閒置審批同樣需要簽名

---

### 🟡 次要不符合 (Minor Non-Conformity)

#### NC-04：維修紀錄狀態自動轉換未記錄稽核軌跡

**位置：** `backend/src/services/equipment.rs:804-809`

**現狀：** 當維修人員點「完成回報」時，後端將 `completed` 自動轉為 `pending_review`，但此自動轉換未記錄到稽核日誌。

**違反條款：**
- **GLP 8.2** — 所有數據變更應可追溯
- **ISO 17025 7.11** — 數據管理應確保完整性

**矯正措施：** 在自動轉換處加入 audit log 記錄

---

#### NC-05：報廢恢復為單人操作，無二次審批

**位置：** `backend/src/services/equipment.rs:1145-1207`

**現狀：** 管理員可單獨將報廢設備恢復為啟用狀態，無需第二人審批。

**違反條款：**
- **ISO 9001 7.5.3** — 文件化資訊的變更應受控

**矯正措施：** 建議加入二次確認或審批流程

---

#### NC-06：維修建立時自動轉狀態，缺少申報人確認

**位置：** `backend/src/services/equipment.rs:738-755`

**現狀：** 新增維修紀錄時，設備自動從 `active` 變為 `under_repair`。建立者可能只是報修，不一定代表設備應立即停止使用。

**建議：** 此行為在操作上可接受（報修即停用是保守策略），但應在 SOP 中明確記載此邏輯。

| 轉換 | 原因 |
|------|------|
| 閒置 → 待修 | 閒置設備若需維修，應先恢復啟用再報修 |

| 報廢 → 閒置 | 報廢設備只能恢復為啟用 |

## 七、閒置審批流程設計建議

既然閒置需要審批，建議參照現有報廢流程的架構：

### 7.1 資料結構

```sql
CREATE TABLE equipment_idle_requests (
    id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    equipment_id          UUID NOT NULL REFERENCES equipment(id),
    request_type          TEXT NOT NULL,  -- 'idle' 或 'restore'
    reason                TEXT NOT NULL,
    applied_by            UUID NOT NULL REFERENCES users(id),
    applied_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    applicant_signature_id UUID REFERENCES signatures(id),
    status                TEXT NOT NULL DEFAULT 'pending',  -- pending/approved/rejected
    approved_by           UUID REFERENCES users(id),
    approved_at           TIMESTAMPTZ,
    approver_signature_id UUID REFERENCES signatures(id),
    rejection_reason      TEXT,
    notes                 TEXT,
    created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 7.2 流程

```
申請閒置                          恢復啟用
  │                                │
  ▼                                ▼
填寫閒置原因 + 簽名              填寫恢復原因 + 簽名
  │                                │
  ▼                                ▼
idle_request(type=idle)          idle_request(type=restore)
status = pending                 status = pending
  │                                │
  ▼                                ▼
通知審批者                       通知審批者
  │                                │
  ▼                                ▼
審批者審核 + 簽名                審批者審核 + 簽名
  │                                │
  ├─ 核准 → 設備 active→inactive   ├─ 核准 → 設備 inactive→active
```
