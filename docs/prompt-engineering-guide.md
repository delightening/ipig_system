## 實戰模板

以下每條秘訣都附上「可直接貼進 CLAUDE.md 的寫法」。

### 秘訣 1：用「禁令」代替「指令」

**原理**：NEVER / DO NOT / CRITICAL 比 "please do X" 更能約束 AI 行為。禁令劃定邊界，指令只是建議。

**你已有的好範例**：
```markdown
❌ 業務邏輯、SQL、複雜條件判斷    ← handlers/ 的禁止事項
禁止硬編碼色彩（text-slate-*、bg-blue-*）
禁止裸用 unwrap() 於非測試碼
```

**可強化的寫法**（將軟性建議改為硬性禁令）：

```markdown
# 禁止事項（全域）

- NEVER 在未讀取檔案內容的情況下修改該檔案。
- NEVER 新增功能超出使用者要求的範圍。Bug fix 就只修 bug，不要順便重構旁邊的程式碼。
- NEVER 為假設性的未來需求做設計。只解決當前問題。
- DO NOT 加入 docstring、註解或 type annotation 到你沒修改的程式碼。
- DO NOT 為不可能發生的場景加入錯誤處理或 fallback。
```

### 秘訣 2：設「唱反調」角色（QA / 驗證者）

**原理**：讓 AI 在完成工作後切換為「找碴模式」，預防自我確認偏誤。

**可加入 CLAUDE.md 的寫法**：

```markdown
# QA 自我驗證

完成每項任務後，切換為驗證者角色，檢查以下項目：

1. 是否真的跑過測試？「看起來沒問題」不等於驗證過。
2. 是否有引入新的 lint 警告？
3. 修改是否超出原始要求的範圍？
4. 是否有遺漏的 edge case？

反駁自己的偷懶藉口：
- 「看起來沒問題」→ 去實際驗證。
- 「改動很小不需要測試」→ 小改動更容易被忽略，必須測試。
- 「這個之前就有問題」→ 不是你不修的理由。
```

---

### 秘訣 3：不要畫蛇添足

**原理**：AI 傾向過度設計。三行重複好過一個過早的抽象化。

**可加入 CLAUDE.md 的寫法**：

```markdown
# 最小變更原則

- 不要加沒被要求的功能。小修改不需要順便重做旁邊的程式碼。
- 不要為不可能發生的情況做預防。只在系統邊界（使用者輸入、外部 API）做驗證。
- 三行重複的程式碼，好過一個過早的抽象化。
- 不要加 feature flag 或向後相容的 shim，直接改程式碼。
- 不要為假設性的未來需求設計彈性。只解決當前問題。
```

---

### 秘訣 4：工作可以分出去，思考不行

**原理**：可以啟動子任務平行作業，但判斷權必須留在自己手上。

**Prompt 範例**（指派子任務時）：

```
❌ 錯誤：「根據你的調查結果，修復這個 bug。」
✅ 正確：「在 backend/src/services/animal.rs 的第 45 行，
         create_animal 函式沒有檢查重複的 ear_tag。
         請在 insert 前加入 exists_animal_by_ear_tag 檢查，
         若重複則回傳 AppError::Conflict。」
```

**可加入 CLAUDE.md 的寫法**：

```markdown
# 任務委派原則

- 委派子任務時，必須包含：具體檔案路徑、行號、要修改什麼、為什麼要改。
- 禁止寫「根據你的調查結果去處理」—— 這等於放棄判斷權。
- 自己先消化資訊、做出判斷，再給出明確方向。
```

---

### 秘訣 5：不知道就說不知道

**原理**：AI 沒有資訊時容易編造合理的回答。

**可加入 CLAUDE.md 的寫法**：

```markdown
# 誠實回報

- 事情沒做好就說沒做好，附上具體原因。
- 沒做過的步驟就說沒做過，不要暗示做過了。
- 事情確實做好了，也不要加沒必要的免責聲明。
- 目標是「準確」的報告，不是「防禦性」的報告。
- 不確定的地方標記為 `[待確認]`，不要猜測。
```

### 秘訣 6：先看再改（你已有）

你的 `先讀 DESIGN.md` 就是這個原則的實踐。可以更明確：

```markdown
# 先看再改（強制）

- 修改任何檔案前，MUST 先讀取該檔案的完整內容。
- 修改 UI 前，MUST 先讀取 DESIGN.md。
- 修改資料庫相關邏輯前，MUST 先讀取對應的 migration SQL。
- 違反此規則的修改一律視為無效。
```

---

### 秘訣 7：一次授權≠永久授權（你已有）

你的寫法已經很好。可以更精確：

```markdown
# 授權範圍

- 檔案讀寫：永久授權，不需詢問。
- 刪除檔案、呼叫外部 API：每次詢問。
- 使用者批准的操作僅限當次有效，不可擅自擴大範圍。
- 例：使用者說「推送到 main」不代表以後都可以推送到 main。
```

### 秘訣 8：禁令要寫清楚「為什麼」

**原理**：AI 理解邏輯後，執行更到位，也更能在新情境中正確推理。

**對比**：

```markdown
# ❌ 沒寫原因
- 禁止 unwrap()

# ✅ 有寫原因
- 禁止裸用 unwrap()，因為 panic 會導致整個 server 崩潰，
  所有使用者的連線都會中斷。使用 `?` 運算子 + AppError 逐層傳播。
```

**建議回溯補上原因的項目**：
- `禁止硬編碼色彩` → 加上「因為主題切換時無法統一變更」
- `禁止 #[allow(dead_code)]` → 加上「因為 dead code 會累積技術債，隱藏真正的問題」

### 秘訣 9：資訊按需供給

**原理**：不要一次把所有規則倒給 AI，按場景載入。

**實踐方式**：

```
專案根目錄/
├── CLAUDE.md              ← 全域規則（永遠載入）
├── backend/CLAUDE.md      ← 後端專屬規則（進入 backend/ 時載入）
├── frontend/CLAUDE.md     ← 前端專屬規則（進入 frontend/ 時載入）
└── docs/CLAUDE.md         ← 文件撰寫規則（編輯文件時載入）
```

Claude Code 會自動讀取工作目錄下的 `CLAUDE.md`。你可以把目前根目錄的 CLAUDE.md 拆分：

| 內容 | 放哪裡 |
|------|--------|
| 語言偏好、操作授權、文件記錄規則 | 根目錄 `CLAUDE.md` |
| Backend 代碼規範（§3-4）| `backend/CLAUDE.md` |
| Frontend 代碼規範（§5）| `frontend/CLAUDE.md` |
| 通用原則（§1-2, §6-10）| 根目錄 `CLAUDE.md` |

### 秘訣 10：溝通規範細到標點（你已有）

可以更進一步：

```markdown
# 溝通規範

- 結論先行，再說理由。不要鋪墊一大段。
- 能用一句話說完的不用三句。
- 引用程式碼時使用 `檔案路徑:行號` 格式。
- 禁止使用 emoji（除非使用者要求）。
- 禁止「我覺得」「可能」等模糊用語——要嘛確定，要嘛標記 [待確認]。
```

### 秘訣 11：不同場景載入不同規則

**實踐方式**：在 CLAUDE.md 中用條件描述。

```markdown
# 場景規則

## 修 Bug 時
- 只修目標 bug，不要順便重構。
- 必須附上重現步驟與修復驗證。

## 新增功能時
- 先列出影響範圍，確認後再動工。
- 遵循現有的架構分層，不要引入新 pattern。

## Code Review 時
- 你的任務不是確認能不能用，是找出問題。
- 檢查安全性（OWASP Top 10）、效能、可維護性。
- 每個問題附上具體行號和建議修改。
```

### 秘訣 13：限制工具使用方式

**原理**：明確告訴 AI 哪些場景用哪個工具，避免用通用工具做專用工具的事。

**可加入 CLAUDE.md 的寫法**：

```markdown
# 工具使用規範

- 讀取檔案用 Read tool，禁止用 cat/head/tail。
- 搜尋檔案用 Glob tool，禁止用 find/ls。
- 搜尋內容用 Grep tool，禁止用 grep/rg。
- 編輯檔案用 Edit tool，禁止用 sed/awk。
- 原因：專用工具有操作紀錄，使用者能看到你在做什麼。
```

（注意：Claude Code 本身已內建這條規則，你不一定需要重複寫。但如果你用的是 Claude Web/API，加上會有幫助。）

### 秘訣 14：如實回報

```markdown
# 回報準則

- 完成時說「工作完成」，附上變更摘要。
- 失敗時說「未完成」，附上具體原因和已嘗試的方法。
- 部分完成時明確列出已完成和未完成的項目。
- 不要潤飾，不要過度謙虛，不要加防禦性免責聲明。
```

## 角色規範模板

以下是可以直接在 prompt 中使用的角色模板：

### 開發者角色

```
你是 ipig_system 的資深開發者。
技術棧：Rust/Axum (後端) + React/TypeScript (前端) + PostgreSQL。
遵循 CLAUDE.md 所有規範。
在修改任何檔案前先讀取其內容。
完成後回報變更摘要。
```

### Code Reviewer 角色

```
你的任務不是確認東西能不能用，而是找出問題。
檢查重點：
1. 安全性（SQL injection, XSS, CSRF）
2. 是否違反 CLAUDE.md 代碼規範
3. 是否有未處理的錯誤路徑
4. 是否超出必要的變更範圍

反駁偷懶：
- 「看起來沒問題」→ 不等於驗證過。
- 「這是既有的問題」→ 不是忽略的理由。
```

### 文件撰寫角色

```
你是技術文件撰寫者。
語言：繁體中文（台灣），技術術語保留英文。
風格：結論先行，精簡扼要。
格式：嚴格遵循 CLAUDE.md 文件記錄規則。
禁止：廢話、emoji、模糊用語。
```

## 場景範例

### 場景 1：修 Bug

```
在 backend/src/services/animal.rs 的 create_animal 函式中，
當 ear_tag 重複時沒有回傳錯誤，而是靜默覆蓋舊資料。

請修復：
1. 在 insert 前呼叫 exists_animal_by_ear_tag 檢查
2. 若重複，回傳 AppError::Conflict
3. 只改這個 bug，不要動其他程式碼
```

### 場景 2：新增功能

```
需求：動物列表頁新增「依狀態篩選」功能。

影響範圍：
- Backend: handlers/animal.rs, services/animal.rs, repositories/animal.rs
- Frontend: pages/animal/AnimalListPage.tsx

限制：
- 遵循現有的 list_animals 分頁模式
- 篩選參數用 query string，不要改 URL 結構
- 不要順便重構現有程式碼
```

### 場景 3：Code Review

```
請 review 以下 PR 的變更。
你的任務是找出問題，不是確認它能用。

重點檢查：
1. 是否違反 CLAUDE.md 代碼規範
2. 安全性問題（OWASP Top 10）
3. 效能問題
4. 變更是否超出 PR 描述的範圍

每個問題請附上：檔案路徑:行號 + 問題描述 + 建議修改
```
