## 1. 取樣說明

### 1.1 流程脈絡
豬博士動物科技為動物試驗受託機構（CRO），內建自有 IACUC 委員會執行審查。流程角色：

- **送件助理**（Evonne / 陳怡均）：外部客戶聯絡窗口 + 委員會秘書。收到客戶計畫書後先做「內部 pre-review」→ 補齊問題 → 再送給 IACUC 委員審查。
- **IACUC 委員**（9 位，匿名標示為 C1–C9）：負責主要審查（每案指定 3 位主審）+ 可選的附議審查。
- **申請人**：外部委託單位或豬博士業務端代填。

這個流程本身就是 R20 想自動化的**兩層預審**：
- 內部 pre-review（助理 → 申請人）= **`CLIENT_REVIEW_PROMPT`** 對應情境
- IACUC 委員審查（委員 → 申請人）= **`STAFF_REVIEW_PROMPT`** 對應情境

## 2. 退件原因分類（MECE）

以下分類按**實際樣本出現頻率由高到低**排列。每類附匿名化案例句式片段（節錄自真實委員或助理回覆）。

### 2.1 交叉引用失效 / 文件內部不一致（最常見）
委員檢查計畫書章節交叉引用時，發現被引用的章節不存在或編號不對。LLM 預審有機會捕捉這類機械性錯誤。

> 真實句式範例（委員 C-吳）：
> * 「『1.1 手術準備…依照上方第 7.2 節進行準備』—— 請問 7.2 節在第幾頁？」
> * 「『6.6 存活手術』所述 7.8 節又在哪裡？」

**樣本數**：≥3 案出現類似問題。

### 2.2 評分量表 / 臨床監測得分門檻定義不清
涉及人道終點（humane endpoint）與術後轉讓判定。委員反覆要求**量化**。

> 真實句式範例（委員 C-徐）：
> * 「6.8 補充説明量表得分意義及於 6.9 補充說明犧牲或轉讓的得分參照…未明確回應我的提問」
> * 「6.8 評估表得分多少，或再應用健康評估表所得的分數，是加總計算還是分開？總分達到多少是安樂死？總分多少又是可以轉讓？」
> * 「我指的量表得分意義為此，而非量表內 0 代表多少，1 代表多少…」

**樣本數**：2 案重複出現同類問題（同一委員對不同案）。反映該委員對「量化的人道終點」特別敏感。

### 2.3 對照組 / 試驗組處置說明不完整
委員追問對照組如何處置、植入物組別差異、每組動物的具體干預。

> 真實句式範例（委員 C-吳）：
> * 「請問是指左側股動脈的 ScCO₂ 去細胞瓣、ePTFE 5 mm 與 SDS 去細胞瓣等植入操作？右側股動脈對照組血管會有何種處置？」

**樣本數**：2 案。

### 2.4 手術後監測項目 / 異常指標定義模糊
委員要求明確列出術後需監測項目與異常定義。

> 真實句式範例（委員 C-張）：
> * 「請說明術後需觀察項目，若出現血循異常，豬博士同仁應注意什麼異常狀況？」

**樣本數**：2 案。

### 2.5 研究目的 vs 動物試驗規範衝突（3R 挑戰）
當計畫目的為「教學 / 訓練」而非「研究假設檢驗」時，委員會質疑是否符合動物試驗規範（Replace / Reduce）。

> 真實句式範例（委員 C-張）：
> * 「本研究的主要目的是在動物模型中，建立…教學平台，提供住院醫師與年輕主治醫師進行血管分離與血管吻合技術之系統化訓練…是藉由豬隻試驗做為練習？這是否符合動物試驗規範？」

**樣本數**：1 案（但屬**重大議題**，可能導致整案重新論證）。

### 2.6 人員名單重複使用 / 資格不更新
同一客戶多次申請但人員名單完全照抄上案，助理會直接擋下。

> 真實句式範例（助理 Evonne）：
> * 「計畫書中的人員名單與先前申請的人員名單完全相同，請務必於實驗前再次確認並進行修正」
> * 「計畫書的簽名日期須請申請人更新」

**樣本數**：3 案。這類是 **pre-review 層**最容易出現的問題，不需審查專業。

### 2.7 試驗物質 / 劑量 / 採血管規格缺失
通常由助理在送審前補齊，不會進委員層。

> 真實句式範例（助理 Evonne）：
> * 「我這邊對於計畫有一些問題需要請申請人補充…」（後接條列式補件清單）
> * 「豬隻體重是否可以放寬至 35-50 kg，以利挑豬」
> * 「依採血時間點分別由前腔靜脈抽取 4.0mL 全血至採血管中…SST 採血管可採至 10ml，故進行以上修正」

**樣本數**：4+ 案。

### 2.8 時程矛盾 / 犧牲時間點定義不清
涉及多次 MRI 追蹤 / 給藥後觀察期的案例。

> 真實句式範例（助理 Evonne）：
> * 「計畫書中寫到『預計在進行同樣療程（連給三周），給完藥物後一個月後再進行一次 MRI，之後進行安樂死犧牲』—— 請問是從第一次給完藥後一個月進行犧牲？還是三次藥物給完後再觀察一個月才進行犧牲？」

**樣本數**：1 案。

### 2.9 重複申請未附上次結果（主要修正 / 第二次申請）
> 真實句式範例（助理 Evonne）：
> * 「委員已回覆審查意見，再請申請人依照內容提供上次的實驗結果與討論改善」

**樣本數**：1 案（樣本不足，但屬明顯 pattern）。

## 3. 審查員語言範本（可作為 prompt few-shot 語料）

以下句式整理自真實委員與助理回信，可直接灌入 system prompt 當 **tone example**：

### 3.1 委員典型質問句（嚴謹、技術導向、短）
1. 「請問 [章節號] 節在第幾頁？[章節號] 所述…又在哪裡？」
2. 「請問是指 [A 操作]、[B 操作] 與 [C 操作] 等植入操作？[對照組] 會有何種處置？」
3. 「請說明術後需觀察項目，若出現 [異常症狀]，同仁應注意什麼異常狀況？」
4. 「…是藉由豬隻試驗做為練習？這是否符合動物試驗規範？」
5. 「[章節] 修正說明未明確回應我的提問…得分多少…是加總計算還是分開？總分達到多少是安樂死？」
6. 「本案無其他意見 / 本案無意見，同意辦理」（正面放行的固定句式）
7. 「我無其他意見」

### 3.2 助理 pre-review 句式（務實、補件導向）
1. 「我這邊對於計畫有一些問題需要請申請人補充，如附件」
2. 「我對於計畫有幾個問題，再請依照內容進行回覆」
3. 「委員已回覆審查意見，再請申請人依照內容進行回覆與補充計畫書」
4. 「計畫書中的人員名單與先前申請的人員名單完全相同，請務必於實驗前再次確認並進行修正」
5. 「計畫書的簽名日期須請申請人更新」
6. 「請將計畫書與修正申請單的簽名頁面簽名後回傳」
7. 「此計畫已審查通過，核准計畫編號為：[PIG-XXXXXX]」

### 3.3 觀察：語言風格
- **極度簡潔**。委員回信通常 1–3 行。不寫客套話。
- **章節號精準引用**。委員習慣 `6.4`、`7.2`、`6.8` 這種明確指標。
- **反問句**遠多於陳述句（委員用反問逼申請人寫清楚）。
- **同意時也極簡**：「本案無意見 / 同意辦理」即放行。

## 4. 對現有 Prompt 的改善建議

### 4.1 現況（截自 `docs/TODO.md` R20-5）

```rust
const CLIENT_REVIEW_PROMPT: &str = r#"
你是一位資深的 IACUC 審查委員，擁有實驗動物科學與獸醫學背景。
你的任務是預審動物實驗計劃書（AUP），幫助計畫主持人在提交前改善內容。
..."#;

const STAFF_REVIEW_PROMPT: &str = r#"
你是一位資深的 IACUC 審查輔助系統，協助執行秘書進行 Pre-Review。
你的任務是標註計劃書中值得注意的地方…"#;
```

### 4.2 缺口 1：缺少「交叉引用一致性檢查」指令

**現象**：最高頻的委員退件原因是「章節交叉引用失效」（見 §2.1）。這是 LLM 擅長的**機械性檢查**，應該強制要求。

**補丁**（可直接貼入 `CLIENT_REVIEW_PROMPT` 與 `STAFF_REVIEW_PROMPT`）：

```
必做檢查：交叉引用完整性
- 逐一列出計畫書中所有形如「依照第 X.Y 節」、「見 X.Y」、「參考 X.Y 節」的章節引用。
- 對每個引用，確認被引用章節是否確實存在於同一份計畫書中。
- 若引用指向不存在的章節，標記為 needs_attention，severity = error。
- 這是最常見的退件原因，不得遺漏。
```

### 4.3 缺口 2：缺少「人道終點量化」稽核指令

**現象**：委員對人道終點定義不清極度敏感，要求**總分門檻明確**（見 §2.2）。現有 prompt 沒有任何關於「量化 humane endpoint」的要求。

**補丁**（`CLIENT_REVIEW_PROMPT`）：

```
必做檢查：人道終點（Humane Endpoint）量化程度
- 若計畫書提及術後評估表、臨床監測表、再應用評估表：
  (a) 確認是否明確寫出 "總分計算方式"（加總？取最大？）
  (b) 確認是否明確寫出 "達到多少分進行安樂死"
  (c) 確認是否明確寫出 "達到多少分可以轉讓給其他實驗"
  (d) 若量表只定義 "0/1/2 代表什麼"，但未定義總分門檻，視為 concern。
- 審查委員常見語句參考：
  "得分多少是加總計算還是分開？總分達到多少是安樂死？"
```

### 4.4 缺口 3：缺少「對照組處置說明」稽核指令

**現象**：委員常質問對照組 / 試驗組每隻動物的具體處置差異（見 §2.3、§2.4）。

**補丁**（`CLIENT_REVIEW_PROMPT`）：

```
必做檢查：組別設計完整性
- 列出計畫書中所有動物分組（試驗組 / 對照組 / 假手術組 ...）。
- 對每個組別，確認是否明確說明：
  (a) 介入措施（植入物種類、劑量、給藥途徑）
  (b) 對照組是否接受假手術？是否留置任何植入物？如無，須明述 "不植入任何物質"。
  (c) 術後觀察項目與時間點
- 若對照組處置缺失或與試驗組對比不清，標記為 concern。
```

### 4.5 缺口 4：缺少「研究目的 vs 3R 原則」挑戰問句

**現象**：當計畫目的是教學訓練時，委員會直接挑戰是否符合動物試驗規範（見 §2.5）。這是 LLM 容易忽略的高階倫理判斷。

**補丁**（`STAFF_REVIEW_PROMPT`，給執行秘書的 AI 輔助）：

```
必做檢查：研究目的與 3R 原則
- 若計畫書的 "研究目的" 或 "研究背景" 包含以下關鍵字：
  "訓練"、"教學"、"住院醫師"、"學習曲線"、"練習"、"技術熟練度"
- 標記為 concern，並附上以下提示給審查秘書：
  "此計畫目的偏向教學訓練而非研究假設檢驗。
   請確認申請人是否已評估 (1) 使用替代模型（Replace）的可行性，
   (2) 可否透過共享手術 / 同一頭動物多次演練減少使用量（Reduce），
   (3) 是否有正式文獻支持豬模型為此訓練的必要載體。"
```

### 4.6 缺口 5：缺少「基本文書完整性」pre-filter（pre-review 層）

**現象**：助理層每次都要重複檢查**人員名單重複、簽名日期過舊、劑量單位遺漏**等低階問題（見 §2.6、§2.7）。這些根本不需要進到委員審查層，浪費委員時間。

**補丁**（`CLIENT_REVIEW_PROMPT` 新增第一階段檢查）：

```
第一階段：文書完整性（必做，先於內容審查）
1. 人員名單檢查：
   - 若系統提供了 "先前同一申請人 / 同一機構的過往計畫書"，比對本次人員名單。
   - 若 100% 相同（姓名、職稱、身份證字號皆同），標記為 needs_attention：
     "人員名單與過往計畫書完全相同，請確認是否需更新。"
2. 簽名日期檢查：
   - 若計畫書版本已更新（修正申請）、但 PI 簽名頁日期仍為舊版日期，標記為 needs_attention。
3. 劑量單位完整性：
   - 所有數值必須帶單位（mg/kg、mL、hr、day）。
   - 採血量必須明述 "每次 X mL" 與 "採血管類型（SST / K2EDTA 等）"。
4. 時程內部一致性：
   - 若計畫書同時提及 "療程週期" 與 "犧牲時間點"，用自然語言推理時序是否衝突。
```

## 5. R20-9 執行路線（下一步建議）

R20-9 待辦原本只寫「收集真實審查意見，對比 AI 預審結果，調整 prompt 提高準確率」——這太粗。基於本次取樣，拆解如下：

### 5.1 Data pipeline
- **階段一（手動，1 天）**：從子瑄 Gmail 匯出 **最近 12 個月** 所有 subject 包含 `APIG-`、`PIG-`、`審查意見`、`修正申請` 的 thread 為 `.eml`。建議用 Gmail 篩選器 + Google Takeout 子資料夾。**不要**透過 API 大量 export；本次搜尋顯示單次查詢容易爆 token 上限。
- **階段二（半自動，1 天）**：用腳本把每個 thread 切成：
  - `protocol_id`（從 subject 正則抽取）
  - `submission_date`（第一封信日期）
  - `round_count`（審查往返次數）
  - `committee_comments`（委員回信全文，去識別化）
  - `applicant_response`（申請人回覆）
  - `outcome`（通過 / 撤案 / 重大修正）
- **階段三**：存成 JSONL，放 `backend/fixtures/iacuc_real_reviews.jsonl`（不進 git，加入 `.gitignore`）。

### 5.2 Labeling（人在迴圈）
- 邀請**怡均**（Evonne）為每筆審查意見標註**缺口類別**（對應 §2 的 9 類）。這是 labeling ground truth。
- 目標：**50 筆標註樣本**即可做 eval baseline。

### 5.3 Eval harness
- 建立 `backend/tests/ai_review_eval.rs`（或獨立 Rust bin）：
  - 輸入：匿名化的計畫書 plaintext（可用 `serialize_protocol_for_ai` 同一套序列化）
  - 預期輸出：標註的缺口類別
  - 指標：
    - **Recall@類別**：AI 是否抓到人類委員抓到的問題
    - **Precision@類別**：AI 警示的問題是否真的重要（避免 false positive 洗版）
    - **Severity alignment**：AI 的 `severity=error/warning/info` 與人類的 major/minor 修正分類是否一致
- 目標基準線：**Recall ≥ 0.7，Precision ≥ 0.6** 再進 R20 release。

### 5.4 Iteration loop
