# 設備維護管理系統 — 設計建議

> 日期：2026-04-05
> 角色視角：器材維運管理人員
> 目的：以 GLP 實驗室業界標準檢視目前系統，找出疏漏並提出改善方案

---

## 一、現況盤點

### 1.1 目前設備狀態 (EquipmentStatus)

| 值 | 中文標籤 | 說明 |
|---|---|---|
| `active` | 啟用 | 設備正常運作中 |
| `inactive` | 停用 | 設備暫停使用（非維修、非報廢） |
| `under_repair` | 維修 | 設備正在維修 |
| `decommissioned` | 報廢 | 設備已報廢 |

### 1.2 目前維修紀錄狀態 (MaintenanceStatus)

| 值 | 中文標籤 | 說明 |
|---|---|---|
| `pending` | 待處理 | 已報修，尚未開始處理 |
| `in_progress` | 進行中 | 維修人員已開始處理 |
| `pending_review` | 待驗收 | 維修完成，等待主管審核簽核 |
| `completed` | 已完成 | 驗收通過 |
| `unrepairable` | 無法維修 | 確認無法修復 |

### 1.3 狀態變更觸發來源（目前）

| 觸發動作 | 設備狀態變化 | 來源 |
|---|---|---|
| 新增維修紀錄 | `active` → `under_repair` | 維修/保養 Tab |
| 維修驗收通過 | `under_repair` → `active` | 維修/保養 Tab |
| 報廢核准 | → `decommissioned` | 報廢 Tab |
| 報廢恢復 | `decommissioned` → `active` | 報廢 Tab |
| **手動編輯設備** | **任意狀態互轉** | **設備 Tab 編輯 Dialog** ⚠️ |

## 二、發現的問題

### 問題 1：設備 Dialog 可直接改狀態 — 繞過流程

**現狀：** 編輯設備 Dialog 有狀態下拉選單，可直接把設備從「啟用」改成「維修」或任意狀態。

**問題：** 這繞過了維修/保養的正式流程（建立紀錄 → 處理 → 驗收簽核），破壞了 GLP 稽核軌跡的完整性。狀態是維修保養流程的結果，不該被手動覆寫。

**建議：** 移除設備編輯 Dialog 中的狀態欄位。設備狀態只能由以下流程驅動：
- 維修/保養流程 → 控制 `active` ↔ `待修`
- 報廢流程 → 控制 → `decommissioned`

### 問題 2：「維修中」語意不精確

**現狀：** 設備狀態 `under_repair` 標籤為「維修」，Dashboard 卡片顯示「維修中」，但只計算設備狀態為 `under_repair` 的數量，不含 `pending`（待處理）的維修紀錄。

**問題：**
- 「維修中」暗示正在維修，但實際上報修後尚未處理的設備也算是不可用
- 使用者預期：已報修 + 送修中 = 都是「不可用，等待修復」的設備

**建議：** 將「維修中」改為「**待修**」，語意涵蓋「已報修等待處理」+「維修進行中」。Dashboard 卡片應計算所有 status = `under_repair`（即有未完成維修紀錄）的設備數量。

### 問題 3：缺少「停用」的明確使用場景

**現狀：** `inactive`（停用）狀態存在，但沒有對應的 UI 流程來觸發。

**問題：** 在 GLP 實驗室環境中，「停用」代表設備暫時不使用但未報修也未報廢（例如：閒置、借出、封存）。目前無明確入口操作此狀態。

**建議：** 保留 `inactive`，但重新命名為「**閒置**」，並在設備 Tab 提供快速操作按鈕（啟用 ↔ 閒置切換），不需走維修流程。

### 問題 4：維修紀錄缺少「送修」階段區分

**現狀：** 維修紀錄有「維修廠商」欄位可選填，但流程上看不出是內修（自行處理）還是外送維修。

**建議（低優先）：** 在維修紀錄中可考慮加入「維修方式」欄位（內修/外送），但不影響設備狀態流程。報修或送修 → 設備都是「待修」。這與你的預期一致：**送修 = 報修 = 待修**。

## 三、建議的設備狀態模型

### 3.1 設備狀態（修正後）

| 值 | 中文標籤 | 說明 | 觸發方式 |
|---|---|---|---|
| `active` | **啟用** | 正常運作 | 維修驗收通過 / 報廢恢復 / 閒置恢復 |
| `inactive` | **閒置** | 暫停使用（非故障） | 設備 Tab 快速操作 |
| `under_repair` | **待修** | 已報修，等待修復 | 新增維修紀錄自動觸發 |
| `decommissioned` | **報廢** | 已報廢除役 | 報廢申請核准 |

### 3.2 設備狀態流轉圖

```
                    ┌──────────┐
         閒置恢復    │          │  快速操作
        ┌──────────▶│   啟用   │◀──────────┐
        │           │ (active) │           │
        │           └────┬─────┘           │
        │                │                 │
        │     ┌──────────┼──────────┐      │
        │     │ 新增維修  │ 快速操作  │      │
        │     ▼          ▼          │      │
   ┌────┴────┐    ┌──────────┐     │      │
   │  待修   │    │   閒置   │─────┘      │
   │(under_  │    │(inactive)│            │
   │ repair) │    └──────────┘            │
   └────┬────┘                            │
        │                                 │
        ├── 維修驗收通過 ─────────────────┘
        │
        ├── 無法維修 → 走報廢流程
        │                    │
        ▼                    ▼
   ┌─────────────────────────────┐
   │           報廢              │
   │     (decommissioned)       │
   └────────────┬───────────────┘
                │
                └── 管理員恢復 → 啟用
```

### 3.3 狀態變更規則（誰能改什麼）

| 操作 | 入口 | 權限 |
|---|---|---|
| 啟用 ↔ 閒置 | 設備 Tab 快速操作按鈕 | `equipment.manage` |
| 啟用 → 待修 | 新增維修紀錄（自動） | `equipment.maintenance.manage` |
| 待修 → 啟用 | 維修驗收通過（自動） | `equipment.maintenance.review` |
| 任意 → 報廢 | 報廢申請核准（自動） | `equipment.disposal.approve` |
| 報廢 → 啟用 | 管理員手動恢復 | `equipment.manage` |

**核心原則：設備 Dialog 只編輯基本資料（名稱、型號、位置等），不觸碰狀態。**

## 四、Dashboard 卡片建議

### 現況

| 卡片 | 計算方式 | 問題 |
|---|---|---|
| 設備總數 (啟用) | `status === 'active'` 的數量 | ✅ OK |
| 維修中 | `status === 'under_repair'` 的數量 | ⚠️ 名稱改為「待修」 |
| 校正/確效/查核紀錄 | 所有校正紀錄總數 | ⚠️ 建議改為更實用指標 |
| 逾期待處理 | 校正逾期數量 | ✅ OK |

### 建議調整

| 卡片 | 計算方式 | 說明 |
|---|---|---|
| 設備總數 (啟用) | `status === 'active'` | 不變 |
| **待修** | `status === 'under_repair'` | 改標籤，邏輯不變（因為建立維修紀錄時已自動改狀態） |
| 校正/確效/查核紀錄 | 所有校正紀錄總數 | 或改為「本月到期」更實用 |
| 逾期待處理 | 校正逾期 + 維修逾期（超過 N 天未處理的待修） | 擴大涵蓋範圍 |

## 五、維修/保養流程建議

### 5.1 維修紀錄狀態（維持不變，語意微調）

| 值 | 中文標籤 | 對應設備狀態 |
|---|---|---|
| `pending` | 待處理 | 設備 = 待修 |
| `in_progress` | 處理中 | 設備 = 待修 |
| `pending_review` | 待驗收 | 設備 = 待修 |
| `completed` | 已完成 | 設備 = 啟用（自動恢復） |
| `unrepairable` | 無法維修 | 設備 = 待修（等待報廢或其他處置） |

### 5.2 完整維修流程

```
使用者報修
  │
  ▼
新增維修紀錄 ──→ 設備自動變「待修」
  │
  ▼
[待處理] ──→ 維修人員接手 ──→ [處理中]
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼              ▼
              填寫維修內容   無法維修         (繼續處理)
                    │         │
                    ▼         ▼
              [完成回報]   [無法維修]
                    │         │
                    ▼         └──→ 通知管理員 → 走報廢?
              [待驗收]
                    │
              ┌─────┴─────┐
              ▼           ▼
          核准+簽名     退回
              │           │
              ▼           ▼
          [已完成]    [待處理] (重新處理)
              │
              ▼
          設備自動恢復「啟用」
```

## 六、具體實作項目（優先順序）

### P0 — 必須修正

| # | 項目 | 說明 |

| 3 | 「維修中」→「待修」標籤更名 | 修改 `EQUIPMENT_STATUS_LABELS.under_repair` = '待修'，Dashboard 卡片標題同步 |

| # | 項目 | 說明 |

## 七、與業界標準對照

### GLP (Good Laboratory Practice) 合規檢查

| 要求 | 目前系統 | 狀態 | 備註 |
|---|---|---|---|
| 設備使用/校正/維修紀錄 | 有完整 timeline | ✅ | — |
| 校正/確效定期執行 | 有年度計畫 + 到期提醒 | ✅ | — |
| 維修需有審核簽核 | 有 pending_review + 電子簽名 | ✅ | — |
| 設備狀態變更可追溯 | 有 equipment_status_logs | ✅ | — |
| 狀態變更需由流程驅動 | **可被手動覆寫** | ⚠️ | P0-1, P0-2 修正後合規 |
| 報廢需審核流程 | 有報廢申請 + 審核 | ✅ | — |
| 停用/閒置管理 | 有 inactive 但無 UI 入口 | ⚠️ | P1-5 改善 |

### ISO 17025 (測試與校正實驗室) 相關

| 要求 | 目前系統 | 狀態 |
|---|---|---|
| 設備唯一識別（序號/型號） | ✅ 有 serial_number, model | ✅ |
| 校正溯源（證書編號、不確定度） | ✅ 有 certificate_number, measurement_uncertainty | ✅ |
| 設備不符合時標識 | ⚠️ 狀態標籤可改善 | P0-3 改善 |

## 八、總結

目前系統在 GLP 合規方面已有很好的基礎（timeline、簽核、status_log），主要缺陷是：

1. **設備狀態可被手動繞過** — 這是最需要立即修正的問題
2. **「維修中」語意不夠準確** — 改為「待修」更符合實際管理需求
3. **閒置管理缺少入口** — 有狀態但沒操作方式

修正 P0 的 3 項後，系統的設備狀態管理即符合 GLP 合規要求。
