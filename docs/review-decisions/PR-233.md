## Breaking Changes 評估（從 PR body changelog）

v26 是 **major breaking release**（changelog 明確標示）：

### 主要 breaking changes

1. **Remove deprecated `initImmediate` option** — `initImmediate` → `initAsync` 重命名（v24 起 deprecated）
2. **Remove legacy `interpolation.format` function** — 老 monolithic format 函數被移除，必須用 Formatter API（`i18next.services.formatter.add()`）
3. **Remove `simplifyPluralSuffix` option** — 舊 v1/v2/v3 相容層 only，影響面小
4. **Remove console support notice** — UI 影響小
5. **Type 重整**：
   - `ExistsFunction` shape 改變（v26.0.8 已 fallback restore，但仍有 type narrowing 變化）
   - `t("key", {} as TOptions)` context 約束變動
   - `addResourceBundle` 6th param 加 `silent` / `skipCopy`

### 安全強化（v26.0.6 security release，採納後**可選擇承擔此版本**）

- ⭐ **CWE-117 防護**：log forging via translation keys / language codes 已修
- ⭐ **ReDoS 防護**：`unescapePrefix` / `unescapeSuffix` regex escape
- ⭐ **Nesting injection**：`escapeValue: false` + nesting block 警告

### 影響面 vs 適配工作量

94 檔案使用 i18next。本 repo `frontend/src/lib/i18n.ts` 是 single point of init。

**主要適配**：
1. `frontend/src/lib/i18n.ts` 檢查 `initImmediate` / `interpolation.format` / `simplifyPluralSuffix` 是否存在
   - 若有 → 改 `initAsync` / 用 Formatter API
2. 94 檔案的 `useTranslation()` / `t(...)` call sites 大多 **runtime 兼容**，但 type 層可能有 TOptions / ExistsFunction 不匹配
3. 跑 `pnpm typecheck` → 預估 5-15 個 type error 集中於少數檔案

**預估工作量：2-4 小時**（i18n.ts init 改動小 + type fix）
