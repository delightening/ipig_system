## Bot Reviews

### CR-1 — CodeRabbit Major (Outside diff, L59-81)

> **簽章建立與 reviewer 更新仍然不是同一個交易。**
> `SignatureService::sign_record()` 這裡吃的是 `&state.db`，不在前面的 `tx` 內；如果後面的 `UPDATE equipment_maintenance_records` 或 `tx.commit()` 失敗，簽章資料已經落地，但 `reviewer_signature_id` 仍可能維持 `NULL`。

**Verify against code**：✅ 真實。

```rust
// PR #241 head 0801bf9f：
let mut tx = state.db.begin().await?;       // L51 — tx 開啟
// ... SELECT FOR UPDATE 在 tx 內 (L52-58)
let signature = SignatureService::sign_record(
    &state.db,                                // L60 — 用 pool 不是 tx ❌
    "maintenance_reviewer", ...
).await?;
sqlx::query("UPDATE equipment_maintenance_records SET reviewer_signature_id ...")
    .execute(&mut *tx).await?;                // L73-79 — UPDATE 在 tx 內
tx.commit().await?;                           // L81
```

**Risk**：sign_record 自帶 commit（獨立 tx），若隨後 UPDATE 或 commit 失敗 →
- 簽章 row 已 persist 在 `signatures` 表
- maintenance record `reviewer_signature_id` 仍 NULL
- `get_signature_infos()` 看得到簽章 → 系統認為「已簽」
- 但 maintenance record 自身查不到 reviewer 連結 → 分裂狀態

**Compliance impact**：21 CFR §11.10(e)(1) 要求簽章與 record 的關聯不可斷裂。此 race 違反。

**Decision**: **DEFER → R29-1**

**Why not ADOPT-now**：
1. 修法需擴 `SignatureService::sign_record_tx(tx, ...)` tx-aware 介面，跨多 callers（`signature/transfer.rs` / `sacrifice.rs` / `observation.rs` 等同模式至少 5 處）
2. 此 PR 是 **critical 合規 hotfix**（補 RBAC + 防覆寫漏洞，有人正在實際使用 sign-maintenance dialog），先讓 RBAC 補上比追求原子性優先
3. tx-aware 重構在多個 sign handler 是更系統性的問題，宜獨立 PR 統一處理（觸及 `services/signature/mod.rs` 介面契約）

**Action**: 加入 `docs/TODO.md` R29 backlog（編號 R29-1），標 MEDIUM (compliance-adjacent)。並 PR description 已將「簽章 + status 變更兩 endpoint 非原子」列為 R3 out-of-scope — 本條（CR-1）是更嚴重的「**同 handler 內**非原子」，需在 backlog 區分。
